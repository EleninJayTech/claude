준비물 & 용어 3분 정리

목표: 시작 전 필요한 것 확인 + 이 가이드에 나오는 단어 이해

준비물 체크

용어 3분 정리

아래 표만 읽으면 이 가이드의 모든 단어를 이해할 수 있습니다.

용어한 줄 설명
CLAUDE.mdClaude가 세션 시작 때 자동으로 읽는 규칙/사실 메모장. 글로벌(내 PC 전체)용과 프로젝트용 두 곳에 둔다.
스킬(Skill)/resume처럼 슬래시(/)로 부르는 저장된 절차. 폴더 안 SKILL.md 파일 하나로 만든다.
서브에이전트특정 역할(조사원·리뷰어 등)을 맡는 부하 Claude. .claude/agents/에 md 파일로 정의하고, 각자 다른 모델을 쓸 수 있다.
모델 별칭opus(가장 똑똑·비쌈) / sonnet(균형) / haiku(빠름·저렴). 별칭은 항상 최신 모델을 자동으로 가리킨다.
opusplan계획은 Opus, 실행은 Sonnet으로 자동 전환해주는 메인 세션 모드. /model opusplan으로 켠다.
deny / allow / askClaude가 실행해도 되는 명령을 정하는 권한 규칙. deny(금지) → ask(물어봄) → allow(허용) 순서로 검사한다.
PROGRESS / DECISIONS세션마다 "오늘 뭐 했나 / 뭘 결정했나"를 남기는 작업 일지 파일. 다음 세션의 Claude가 읽고 이어서 일한다.
@경로/파일Claude에게 "이 파일 읽어" 하고 정확히 지목하는 문법. 예: @docs/PROGRESS.md
왜 이걸 셋업하나요?Claude Code는 세션이 끝나면 대화를 잊습니다. 그래서 ① 기록 파일(PROGRESS/DECISIONS)로 기억을 이어주고, ② 권한 규칙으로 사고를 막고, ③ 모델 분담으로 비용을 아끼는 게 이 가이드의 전부입니다.

글로벌 셋업 — PC당 딱 1회

목표: 내 PC의 모든 프로젝트에 적용될 공통 규칙·권한·스킬 만들기 (약 10분)

STEP 1폴더 만들기

PowerShell을 열고 아래를 그대로 붙여넣으세요. (<내계정>은 자동으로 처리됩니다)

PowerShell
cd $env:USERPROFILE
New-Item -ItemType Directory -Path .claude/skills/resume -Force
New-Item -ItemType Directory -Path .claude/skills/wrap -Force

STEP 2글로벌 CLAUDE.md 만들기 (행동 규칙)

에디터로 아래 경로에 새 파일을 만들고 내용을 붙여넣으세요.

C:/Users/<내계정>/.claude/CLAUDE.md
# CLAUDE.md (Global)
## 1. Think Before Coding — 가정 명시, 불확실하면 질문, 해석 갈리면 제시.
## 2. Simplicity First — 요청 이상 금지, 1회용 코드 추상화 금지.
## 3. Surgical Changes — 범위 외 코드 손대지 않기, 기존 스타일 따르기.
## 4. Goal-Driven Execution — 수정 전 검증 기준 먼저, 다단계는 짧은 계획.
## 5. Output Style — 간결하게, 전체 파일 재출력 금지(변경분만).
## 6. File Reference — 파일은 @경로/파일명으로 참조.
## 7. Compaction Priority — 압축 시 보존: 수정 파일·대기 작업·최근 결정·미해결.
## 8. Secret Safety — .env·키·인증서 읽거나 cat/type/Get-Content 금지, 시크릿은 환경변수로.

STEP 3글로벌 settings.json 만들기 (권한 + 알림)

C:/Users/<내계정>/.claude/settings.json
{
  "autoMemoryEnabled": true,
  "attribution": { "commit": "", "pr": "" },
  "permissions": {
    "deny": [
      "Read(**/.env)", "Read(**/.env.*)", "Read(**/*.pem)", "Read(**/*.key)", "Read(**/*.p12)",
      "Bash(cat *.env)", "Bash(cat *.env.*)", "Bash(cat *.pem)", "Bash(cat *.key)",
      "Bash(type *.env*)", "Bash(Get-Content *.env*)", "Bash(gc *.env*)",
      "Read(**/node_modules/**)", "Read(**/.next/**)", "Read(**/dist/**)", "Read(**/build/**)",
      "Read(**/out/**)", "Read(**/target/**)", "Read(**/.gradle/**)", "Read(**/vendor/**)",
      "Read(**/*.jar)", "Read(**/*.class)", "Read(**/*.log)"
    ]
  },
  "hooks": {
    "PreCompact": [ { "matcher": "auto", "hooks": [
      { "type": "command", "command": "echo \"[알림] 곧 컨텍스트 압축 - /wrap 권장\" 1>&2", "async": true } ] } ],
    "SessionStart": [ { "matcher": "compact", "hooks": [
      { "type": "command", "command": "echo '압축 후 복구: docs/PROGRESS.md 최상단과 PROJECT_PLAN.md를 다시 읽어라.'" } ] } ]
  }
}
이 deny 규칙이 하는 일 (2026-07-20 재검증 기준)① .env·인증서를 Read 도구와 Bash 안의 파일 명령(cat/type 등)에서 차단합니다. Read deny는 같은 경로의 Edit도 차단(v2.1.208+). ② 단, python/node 스크립트가 파일을 직접 여는 것까지는 못 막습니다 — 진짜 비밀은 아예 레포에 두지 않는 게 근본 해법입니다(부록 A). ③ OS 샌드박스는 macOS·Linux·WSL2에서 지원(/sandbox) — 네이티브 Windows만 미지원이라 Windows에선 규칙이 더 중요합니다.

STEP 4/resume · /wrap 스킬 만들기

세션 시작·종료 버튼입니다. 파일 2개를 만드세요.

C:/Users/<내계정>/.claude/skills/resume/SKILL.md
---
name: resume
description: 세션 시작 시 git status로 미커밋 작업 먼저 확인 후 CLAUDE.md·docs/PROGRESS.md 최상단·PROJECT_PLAN.md를 읽고 지난 상태·다음 작업 보고.
---
# /resume — 먼저 git status·브랜치로 미커밋(진행 중) 작업 발견 → CLAUDE.md, docs/PROGRESS.md(최상단), PROJECT_PLAN.md를 읽고 "지난 X, 다음 Y?" 보고.
C:/Users/<내계정>/.claude/skills/wrap/SKILL.md
---
name: wrap
description: 세션 종료 시 docs/PROGRESS.md 최상단에 오늘 작업 append, 새 결정 DECISIONS, PROJECT_PLAN 체크박스, 미커밋 경고, 변경 파일 보고.
---
# /wrap — docs/PROGRESS.md 최상단 append(작업/결정/다음/미해결) + 새 DEC + PROJECT_PLAN 체크박스. 미커밋이면 경고. 변경 파일 보고.

확인제대로 됐는지 검증

확인 방법아무 폴더에서 claude 실행 → / 만 입력 → 자동완성 목록에 /resume과 /wrap이 보이면 성공! 안 보이면 부록 B(문제 해결)로.

프로젝트 셋업 — 레포마다 1회

목표: 내 프로젝트에 기록 시스템 + 권한 + 충돌 방지 심기 (약 15분)

STEP 0내 환경은 어느 유형? (30초 판별)

유형이런 경우할 일
🅱️ 단일 repogit repo가 1개 (신입은 대부분 여기!)아래 STEP 1~5 그대로
🅰️ 통합 멀티레포여러 repo를 함께 작업각 repo에 STEP 1~5 반복 + 아래 접기 참고
➕ MSA 단위분할여러 팀원이 같은 repo를 동시에 편집STEP 1~5 + 아래 접기 참고

이 레벨은 단일 repo 기준으로 진행합니다. 멀티레포·팀 협업은 하단 접기에 요약해 두었습니다.

STEP 1docs 기록 파일 3종 만들기

프로젝트 루트에서 PowerShell로:

PowerShell — 프로젝트 루트에서
New-Item -ItemType Directory -Path docs -Force
New-Item -ItemType File -Path docs/PROJECT_PLAN.md, docs/PROGRESS.md, docs/DECISIONS.md -Force

세 파일에 시작 내용을 붙여넣으세요.

docs/PROJECT_PLAN.md
# PROJECT_PLAN (마스터 로드맵)
## 현재 Phase
- [ ] Phase 1: (첫 목표를 적으세요)
## 다음 Phase
- [ ] Phase 2: ...
docs/PROGRESS.md
# PROGRESS (최신이 맨 위 · append 전용)
<!-- 양식: - [단위][상태] 설명 — @작성자 (브랜치) -->
- [셋업][Done] 프로젝트 기록 시스템 초기화 — @나 (main)
docs/DECISIONS.md
# DECISIONS (결정 기록 · append 전용)
<!-- id 양식: DEC-날짜-작성자 (동시 발번 충돌 방지) -->
- DEC-20260703-me: 세션 기록은 /resume·/wrap 루틴으로 관리한다.

STEP 2프로젝트 CLAUDE.md 만들기 (사실 + 기록 규칙)

루트에 CLAUDE.md를 만들고, 윗부분은 내 프로젝트에 맞게 수정, 아래 "문서·기록 규칙" 블록은 그대로 두세요.

<프로젝트루트>/CLAUDE.md
# CLAUDE.md (이 프로젝트의 사실)
## 스택
- (예: Java 21, Spring Boot 4, MySQL 8 — 내 프로젝트에 맞게 수정)
## 빌드/테스트 명령
- (예: ./gradlew test)
## 제약
- (예: DB 스키마 변경은 Flyway로만)

## 문서·기록 규칙 (Claude가 자동 적용)

### 작업 기록 구조
- 기록(PROGRESS·DECISIONS)은 docs/에 둔다. 마스터 PROJECT_PLAN.md만 docs/ 루트.
- 기록 항목 양식: [단위][상태(Done/Pending/Blocked)] 설명 — @작성자 (브랜치). 최상단 append.

### 동시편집 충돌
- PROGRESS·DECISIONS는 append 전용(기존 줄 수정 금지). .gitattributes의 merge=union이 동시 append를 자동 합침.
- DECISIONS id는 날짜+작성자 포함(예: DEC-20260703-min).

### 세션 워크플로 규율
- append-only(과거 수정·삭제 금지). 결정이 바뀌면 새 DEC + 기존에 "Superseded by DEC-…" 표시.
- /resume: 먼저 git status·브랜치로 미커밋(진행 중) 작업 발견 → 그다음 PROGRESS 최상단 + PROJECT_PLAN 현재 Phase + 최근 DEC 3건.
- /wrap: PROGRESS append + 새 DEC + PROJECT_PLAN 체크박스 갱신 + 미커밋이면 경고.

### 길이 관리
- PROGRESS가 약 800줄/분기 경계를 넘으면 가장 오래된 분기를 docs/archive/로 옮기고 활성 파일 맨 아래 포인터 한 줄.

### 공유 vs 개인 / 시크릿
- 공유(커밋): CLAUDE.md·.claude/skills·.claude/settings.json·.gitattributes·docs/.
- 개인(커밋 금지): .claude/settings.local.json·CLAUDE.local.md.
- 시크릿(.env·키·비밀번호)은 읽지도 커밋하지도 않는다.

### 토큰 참고
- docs/는 자동 선로딩되지 않고 필요 시만 읽힌다. 분할은 토큰이 아니라 정리·타겟 정확도용.

STEP 5커밋해서 팀과 공유

PowerShell — 프로젝트 루트에서

git commit -m "chore: Claude Code 세션 기록 시스템 셋업"
🅰️ 멀티레포 / ➕ 팀 동시편집이라면? (요약)
  • 멀티레포: 레포를 물리적으로 합치지 마세요(독립 git이 깨짐). 각 repo에 위 STEP 1~5를 반복하고, 여러 repo가 한 폴더 아래 있으면 상위 폴더에서 claude 실행 시 전부 인식됩니다. 흩어져 있으면 claude --add-dir ../repoB. 커밋은 항상 repo별로 따로.
  • 팀 동시편집(MSA): docs/<단위>/PROGRESS.md처럼 단위별 폴더로 쪼개고, /resume·/wrap이 "PWD → 브랜치 → 질문" 순서로 대상 단위를 판별하게 합니다. 폴더 분할은 정리용이고, 실제 충돌 방지는 위의 merge=union이 담당합니다.

매일 루틴 — 이 6줄이 전부

목표: 매일 이 순서만 지키면 기록이 끊기지 않는다

프로젝트 폴더에서 claude 실행
→ git pull  # 동료 기록 동기화
→ /resume  # 어제 상태 복구 (미커밋 먼저 확인)
→ @경로/파일 작업  # 작게 쪼개서
→ /wrap  # 오늘 기록 남기기
→ git add · commit · push  # 반드시!
가장 흔한 실수: 기록만 하고 커밋 안 하기커밋하지 않은 기록은 다음 세션의 /resume이 git status로만 겨우 발견합니다. /wrap 후 커밋까지가 한 세트입니다.

상태 뱃지 3종 (기록 양식)

뱃지의미
🚀 [Done]PR/병합 가능한 독립 완결
⏳ [Pending]정상 빌드, 잔여 스펙 있어 이어서
🚨 [Blocked]컴파일 에러/외부 의존 교착으로 동결
기록 한 줄 예시 (PROGRESS.md)
- [게시판][Done] 카테고리 트리 추가 — @작성자 (feature/board)
4

모델 자동 분담 — 비용은 줄이고 품질은 지키기

목표: 어려운 일=Opus, 잘 정의된 일=Sonnet/Haiku 로 자동 분담하는 서브에이전트 팀 만들기

핵심 아이디어 한 줄비싼 모델(Opus)은 판단·설계·검증에만 쓰고, 싼 모델(Sonnet/Haiku)에게는 잘 정의된 실행만 시킵니다. 어디가 어려운지 분류(트리아지)와 최종 검증만 잘하면 비용이 크게 떨어집니다.

단계 → 모델 배정표 (기본값)

단계에이전트모델이유
1. 구조 조사(survey)surveyorsonnet기계적·반복 → 저렴하게
2. 난이도 분류 ⭐triageopus여기서 틀리면 하류 전체 오염 (최고 레버리지)

  1. 문서 고정 핸드오프 — 단계 산출물을 파일로 남기고 다음 단계는 @파일로 읽는다. 대화 요약으로 넘기지 않는다(컨텍스트 손실이 최대 실패 원인).
  2. 트리아지는 상위 모델 — "무엇이 어렵고 쉬운가" 분류가 최고 레버리지.
  3. 검증 스테이지 필수 — Opus diff 리뷰 + 정답 대조. "검증기가 모델보다 중요."

WAVE 특화 팁

  • 3,000줄+ 거대 파일(camp_edit.php) 분석 → opus[1m] (1M 컨텍스트)로.
  • 高모듈 구현: spec이 탄탄하면 opusplan, camp 상태머신처럼 실행 중 난관이 예상되면 /advisor.
  • effort 단계별 배정: triage·reviewer는 high 이상 유지, parity-tester는 low로 낮춰도 무방.

deny 규칙이 어디까지 막고, 어디부터 못 막는지 (2026-07 검증 기준)

상황deny가 막나?
Read 도구로 .env 읽기✅ 막음
Bash 안의 파일 명령(cat/type/head 등)으로 읽기✅ 감지·차단 (v2.0.22+ 개선. 단 규칙을 넣어야 함 — cat 계열은 원래 무프롬프트 내장 명령)
python/node 스크립트가 파일을 직접 open🚫 못 막음 (서브프로세스 우회)
OS 샌드박스로 차단✅ macOS·Linux·WSL2 지원(/sandbox) · 🚫 네이티브 Windows만 미지원 → WSL2 또는 규칙 기반
그래서 결론은 하나진짜 비밀번호·API 키·인증서는 작업 레포에 아예 두지 않습니다(별도 vault·환경변수·_secure_config). deny는 부분 안전망이고, 시크릿 분리가 근본 해법입니다.
  • 규칙 우선순위: deny → ask → allow 순서로 첫 매치가 승리. 넓은 deny는 좁은 allow보다 항상 우선(deny에 예외를 뚫을 수 없음).
  • PowerShell 사용자: cat만 막으면 type·Get-Content·gc로 뚫립니다 — 레벨 1·2의 settings.json에 이미 포함해 두었습니다.
  • 행동 규칙으로 이중 방어: 글로벌 CLAUDE.md #8("시크릿 읽기 금지")이 규칙 레벨을 보완합니다.
