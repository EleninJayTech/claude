이게 뭐 · 왜 · 결과

사용법: 원본 md를 프로젝트 루트에 넣고 "이 플레이북을 이 프로젝트에 적용해줘"

내용
무엇어려운 추론은 상위 모델(예: Opus), 잘 정의된 실행은 하위 모델(예: Sonnet/Haiku)로 자동 분담하는 작업 파이프라인을 이 프로젝트에 맞게 세팅
왜품질은 지키고 토큰/비용은 줄인다. 프로젝트마다 스택·성격이 다르므로 고정 템플릿이 아니라 학습+질문으로 맞춘다
결과.claude/agents/(모델별 서브에이전트) + docs/(난이도 기준·핸드오프 규약)이 만들어지고 1단계부터 실행 가능

적용을 지시하면 Claude는 ①프로젝트를 스스로 학습 → ②모르는 것만 질문 → ③맞춤 구성 생성 → ④승인 후 착수 순서로 움직입니다(§2).

1

불변 원칙 5 — 모든 프로젝트 공통

프로젝트가 무엇이든 이 5가지는 항상 지킨다

  1. 문서 고정 핸드오프 — 단계 산출물을 파일로 남기고 다음 단계는 @파일로 읽는다. 대화 요약으로 넘기지 않는다(컨텍스트 손실이 최대 실패 원인).
  2. 트리아지는 상위 모델 — "무엇이 어렵고 무엇이 쉬운가" 분류가 최고 레버리지. 여기서 틀리면 하류 전체가 오염된다.
  3. 검증 스테이지 필수 — 상위 모델의 diff 리뷰 + 정답 대조(테스트/파리티). "검증기가 모델보다 중요."
  4. 난이도 기반 라우팅 — 강한 모델=추론(분석/설계/엣지), 약한 모델=실행(정형/반복). 잘 정의된 것만 싸게.
  5. 자동화 천장 인정 — 완전 자동은 없다(~70~85%). 계획·비기능요구(성능/보안/구조)·최종 리뷰는 사람이 소유.
2

Claude 실행 절차 — "적용해줘" 하면 이 순서

STEP 1~4: 학습 → 질문 → 생성 → 승인

STEP 1프로젝트 자동 학습 (스스로)

코드/설정으로 알 수 있는 건 질문하지 않고 직접 확인합니다:

  • 스택/런타임: package.json·pom.xml·build.gradle·composer.json·requirements.txt·go.mod·Cargo.toml 등 감지 → 언어·프레임워크·버전
  • 빌드/테스트/CI: 명령, 기존 테스트 유무, CI 설정
  • 구조/도메인: 디렉터리 트리, 주요 모듈, 엔트리포인트, README·CLAUDE.md·docs
  • 규모/성격: 파일 수·크기, 사내툴/외부서비스/라이브러리 여부
  • 경계/시크릿: 건드리면 안 되는 영역, 시크릿 위치(읽지 않는다)
  • git: 브랜치·원격·미커밋 여부

STEP 2부족분 질문 (모르는 것만)

코드로 확인 불가능한 것만 §3의 질문을 한 번에 묶어, 각 질문에 추천 기본값과 함께 묻습니다. 이미 파악한 건 "이렇게 이해했다"로 확인만.

STEP 3이 프로젝트에 맞춘 구성 생성

  • (a) 난이도 rubric을 이 프로젝트 기준으로 작성 → docs/난이도-매트릭스.md 골격
  • (b) 에이전트 로스터 + 모델 배정 → .claude/agents/에 생성 (§4 템플릿을 이 스택에 맞게: tools·빌드/테스트 명령·용어)
  • (c) 오케스트레이션 순서 + 핸드오프 문서 규약(어느 문서에 무엇) → docs/에 기록

STEP 4확인 후 착수

구성 요약을 보고하고 승인을 받은 뒤 1단계(survey/분석)부터 실행. 승인 없이 대량 변경하지 않는다.

3

적응 규칙 + 질문 템플릿

프로젝트마다 "알아서 채우는" 변수들

적응 규칙 (Claude가 스스로 조정)

프로젝트 변수어떻게 조정
지배적 활동 (레거시 전환 / 신규 개발 / 리팩터 / 감사·리뷰 / 버그헌팅)로스터 형태를 바꾼다. 전환·리팩터 = survey→triage→analyst→implement→review · 신규 개발 = spec→implement→review · 감사 = finder→verifier
스택/빌드에이전트 tools와 빌드·테스트·린트 명령을 그 스택에 맞춤 (./gradlew test·npm test·pytest)
규모크면 슬라이스를 잘게·병렬로, 작으면 단일 흐름. 병렬 도구: 파일을 고치는 에이전트 병렬은 frontmatter isolation: worktree(각자 격리 checkout — 충돌 원천 차단), 사람 주도 세션 병렬은 claude --worktree <이름>. 초대형 코드베이스는 1M 컨텍스트 활용 — 단 Anthropic API에선 Sonnet 5·Opus 4.8이 기본 1M이라 suffix 불필요(구독 Max·Team·Enterprise도 Opus 자동 1M). [1m] suffix(opus[1m]·opusplan[1m])는 게이트웨이·구형 모델·비자동 티어에서만 의미
성격 (사내툴 / 외부 서비스 / 라이브러리)검증의 NFR 가중치 조정 (외부 서비스=보안·성능↑, 라이브러리=API 호환·문서↑)
난이도 高 정의그 스택의 위험 요소로 채움. 예) 암호화·인증·동시성·상태머신·외부 연동 규약·거대·강결합 파일·성능 임계 경로 / 易 = CRUD·조회·정형 변환·보일러플레이트

질문 템플릿 6 (추천 기본값 포함)

  1. 목적/활동: 이 작업의 목적은? (전환 / 신규 기능 / 리팩터 / 감사 / 버그수정)
  2. 타깃/제약: 목표 스택·버전? 반드시 지킬 제약(DB 스키마·외부 API·프로토콜)?
  3. 범위/우선순위: 먼저 할 영역과 제외 영역? (추천: 저위험·읽기중심부터)
  4. 모델/비용: 쓸 수 있는 모델? (opus/sonnet/haiku) 비용 상한·분담 선호? (추천 기본값: §5)
  5. 검증(정답): 기존 테스트/CI 유무? "정답(oracle)"은 무엇? (기존 동작 재생·골든테스트·수동확인)
  6. 팀/기록: 다중 개발자? 작업 기록 위치? (추천: docs/ append 전용)
4

에이전트 로스터 템플릿 — Claude가 <…>를 채운다

배치: 팀 공유면 <repo>/.claude/agents/, 개인 실험이면 ~/.claude/agents/

frontmatter 옵션 2개① 강도 차등: effort:(low~max) — 세션 effort를 그 에이전트 실행 중에만 오버라이드(공식 확정 키). ② 병렬 안전: 파일을 고치는 에이전트(implementer 등)를 병렬로 돌릴 땐 isolation: worktree — 각자 격리된 checkout에서 작업해 파일 충돌 원천 차단(변경 없으면 자동 정리).

아래는 "전환/리팩터형" 기본 로스터입니다. 신규개발형이면 survey/triage를 빼고 spec-writer부터 시작합니다. 모델 값은 별칭(opus·sonnet·haiku) 또는 전체 ID.

.claude/agents/surveyor.md
---
name: surveyor
description: <프로젝트> 구조 survey 전문. 모듈·의존성·엔트리포인트를 목록화. 넓은 조사에 적극 사용. 파일 수정 안 함.
tools: Read, Grep, Glob, Bash
model: sonnet
---
넓고 기계적으로 훑어 docs/survey/<area>.md 에 표·목록으로 남긴다. 판단·설계는 하지 않는다.
.claude/agents/triage.md
---
name: triage
description: 난이도 분류 전문. survey + rubric으로 모듈별 난이도(高/低)·근거·권장모델·순서를 매긴다.
tools: Read, Grep, Glob
model: opus
---
입력 @docs/survey/*.md, rubric(<高 정의>)에 따라 판정 → docs/난이도-매트릭스.md. 애매하면 高로 올리고 이유를 남긴다.
.claude/agents/analyst-hard.md — 쌍둥이 analyst-light.md는 model: sonnet만 변경
---
name: analyst-hard
description: 高난이도 모듈 상세 분석·<타깃 스택> 설계. 구현 spec(성공기준·샘플 입출력·위험) 작성.
tools: Read, Grep, Glob
model: opus
---
입력 @docs/난이도-매트릭스.md 해당 모듈 + 관련 코드 → docs/spec/<module>.md. 기존 버그 보존 여부는 "사람 확인 필요" 표시.
.claude/agents/implementer.md
---
name: implementer
description: spec 기반 <타깃 스택> 구현 전문(정형 모듈). spec 준비되면 적극 사용.
tools: Read, Write, Edit, Bash
model: sonnet
---
입력 @docs/spec/<module>.md(대화 요약 아님). 성공기준·샘플 만족하게 구현, 범위 밖 손대지 않음. 변경 요약 보고.
.claude/agents/reviewer.md
---
name: reviewer
description: 변환/구현 코드 diff 검증. 타입·널·관용구·상태·NFR(성능·보안) 관점. 구현 직후 적극 사용. 읽기 전용.
tools: Read, Grep, Glob, Bash
model: opus
---
spec과 diff를 대조해 의미 오역을 잡는다. 코드는 고치지 않고 무엇이/왜/어떻게 지적만.
정확성·요구사항에 영향 주는 것만 지적하고 스타일 취향은 제외한다(빈틈을 찾으라면 멀쩡한 코드에서도 찾아내므로 — 과설계 방지).
.claude/agents/parity-tester.md
---
name: parity-tester
description: 정답 대조(테스트/재생) 실행. 빠르고 저렴하게 통과/차이만 보고.
tools: Bash, Read
model: haiku
---
<테스트 명령>으로 검증하고 diff만 보고. 원인 분석은 상위 단계로.
5

기본 모델 배정 — 사용자가 다르게 원하면 덮어씀

단계별 기본값 + 하이브리드 옵션 2종 + 비용 제2축

단계기본 모델이유
survey / 정형 실행 / 테스트sonnet / haiku기계적·반복 → 싸게
난이도 분류(triage)opus최고 레버리지
상세 분석·설계低 sonnet / 高 opus난이도별
高난이도 구현opusplan(메인 세션) 또는 /advisoropusplan=계획 Opus/실행 Sonnet 자동 · advisor=실행 중에도 Opus 온디맨드 자문
리뷰opus의미 오역·NFR 검출
opusplan 주의opusplan은 메인 세션 모드 별칭(Plan Mode=Opus → 실행=Sonnet)입니다. 서브에이전트 frontmatter의 model: 값으론 못 씁니다(sonnet/opus/haiku/fable/전체 ID/inherit만 유효, 기본값 inherit). 어려운 구현은 서브에이전트 대신 메인 세션 /model opusplan으로. availableModels 상호작용(v2.1.205+): Opus 최신이 제외돼도 허용된 최신 Opus로 plan 실행, 모든 Opus가 제외된 경우에만 Sonnet 유지.
opusplan vs /advisoropusplan은 plan 경계에서 모델을 바꾸고, /advisor는 실행 중 주모델이 막힐 때 상위 모델에 자문(모델이 자문 시점을 스스로 판단, 전체 대화를 advisor가 읽음)합니다. 공식 측정(블로그 "The advisor strategy"): Sonnet+Opus advisor = SWE-bench Multilingual +2.7%p·태스크당 비용 −11.9%(Sonnet 단독 대비), BrowseComp에선 Haiku+Opus advisor 41.2%(단독 19.7%). 계획이 명확한 정형 전환엔 opusplan, 실행 중 난관이 예상되는 탐색적 작업엔 advisor가 유리합니다. 제약(2026-07 현행): 실험적·Anthropic API 전용(Bedrock/Vertex/Foundry 불가). 설정: /advisor opus·advisorModel·--advisor. advisor는 주모델과 같거나 더 유능해야 하며, Fable 5는 현재 advisor 역할로는 제공 안 됨.
비용 제2축 — effort모델 선택과 별개로 /effort(low~max)로 추론 강도를 조절할 수 있습니다. "라우팅으로 모델을, effort로 강도를" 2축 운용이 최종 형태입니다. 에이전트별 차등은 frontmatter effort: 키(공식 확정)로 겁니다. 다만 강도를 프로젝트 설정에 고정하지는 마세요(effortLevel 비권장) — 세션 단위 단일값이라 작업별 최적이 안 되고, 서브에이전트 차등에도 쓸 수 없습니다(그건 에이전트 frontmatter effort: 몫). 기본값 + 필요할 때 /effort 가 최적입니다. ⚠️ 세션이 계속 xhigh로 시작한다면 모델 기본값이 아니라 프로젝트 .claude/settings.json의 effortLevel 때문일 가능성이 큽니다(Project > User, 경고 없음) — 통합마스터 D-6 참조.
대량 정형 전환의 대안 — claude -p 팬아웃파일 수백 개짜리 마이그레이션은 세션 안 서브에이전트보다 헤드리스 루프가 더 통제 가능합니다(파일당 독립 컨텍스트 + --allowedTools로 권한 좁게). 순서: 대상 파일 목록 생성 → 2~3개로 프롬프트 튜닝 → 전체 루프 실행. 상세는 02 진단·확장활용 가이드 §8.
6

착수 전 자기 점검

Claude(그리고 나)가 시작 전에 확인할 5가지

A

주의 — 기대치 관리

이 체계로도 안 되는 것들을 미리 알기

  • 완전 자동 아님(~75% 천장): 숨은 비즈로직·아키텍처·보안 엣지는 사람이 판단.
  • "almost right" 함정: 어려운 걸 싼 모델로 밀면 재작업이 더 비쌀 수 있음 → rubric으로 걸러 高는 상위 모델.
  • 비기능요구(성능/보안/구조)는 자주 누락 → reviewer가 명시 점검.
  • "N% 자동 보존" 류 벤더 수치 불신 → 정답 대조로 직접 검증.
  • 내장 라우팅과 중복 주의: Claude Code는 탐색에 내장 Explore 서브에이전트를 자동 사용. Haiku 고정은 구정보 — 현행은 메인 세션 모델 상속(Opus 상한). 싸게 고정하려면 Explore 이름의 커스텀 에이전트를 model: haiku로 정의(내장을 덮어씀). 단순 코드 탐색까지 커스텀 surveyor로 감싸면 과설계 — surveyor는 "문서 산출물이 필요한 survey"에만. (내장 Explore·Plan 끄기: CLAUDE_CODE_DISABLE_EXPLORE_PLAN_AGENTS=1, v2.1.198+)
B

모델 라인업 참고 (2026-07-20 재검증)

별칭이 자동 추적하므로 §5 배정표는 그대로 유효

  • 별칭: haiku / sonnet / opus / fable / best / default. best = 조직에 Fable 5 접근권 있으면 Fable 5, 없으면 최신 Opus. default는 오버라이드를 지우고 계정 기본값 복귀(별칭 아님).
  • 별칭 해석은 프로바이더별로 다름: Anthropic API 기준 opus=Opus 4.8(v2.1.154+)·sonnet=Sonnet 5(v2.1.197+). Bedrock/Vertex/Foundry는 더 낮은 버전일 수 있음.
  • Claude Fable 5(v2.1.170+): 최상위 모델, 기본값 아님 → /model fable로 선택. 장시간 자율 세션·자체 검증 강화형 — "단계가 아니라 결과를 기술"하는 프롬프트에 최적. 안전 분류기 발동 시(사이버보안·바이오 등) Opus 4.8로 자동 폴백(/model fable로 복귀).
  • 컨텍스트: Anthropic API에선 Fable 5·Sonnet 5·Opus 4.8·4.7이 기본 1M. 구독(Max·Team·Enterprise)은 Opus 자동 1M 업그레이드. [1m] suffix는 그 외 경로용.
  • 기본 모델(계정별): Max·Team Premium·API = Opus 4.8 / Pro·Team Standard·Enterprise 구독 = Sonnet 5.
  • 버전 핀: 전체 ID(claude-opus-4-8) 또는 ANTHROPIC_DEFAULT_OPUS_MODEL/…SONNET…/…HAIKU…/…FABLE_MODEL 환경변수.
  • 가용성 폴백: --fallback-model sonnet,haiku 또는 fallbackModel 설정(과부하 시 체인 순서 대체, 최대 3개).
