문서 저장소 사용법

이 저장소의 md 문서들은 읽는 문서가 아니라 "실행되는 문서"입니다. 프로젝트에 넣고 Claude에게 "적용해줘"라고 말하면, Claude가 문서를 읽고 알아서 환경을 구성합니다. 개발을 배우는 중이라면 이 페이지 하나로 전체 그림을 잡을 수 있습니다.

1핵심 개념 — "드롭인 문서"

복사해 넣고, 한 문장 말하면 끝

보통 개발 가이드는 사람이 읽고 손으로 따라 합니다. 이 저장소의 문서는 다릅니다 — Claude가 읽고 Claude가 따라 합니다.

1. 문서(md)를 내 프로젝트 폴더에 복사
2. Claude Code 실행 후: "00_통합-설치.md 를 학습해서 이 프로젝트에 적용해줘"
3. Claude가 환경을 스스로 파악 → 선택 목록 제시 → 체크한 것만 설치 → 검증 결과 보고

이렇게 하면 설정 파일(CLAUDE.md, .claude/, docs/)을 직접 만드는 법을 몰라도, 검증된 구성을 어떤 PC·어떤 프로젝트에든 똑같이 재현할 수 있습니다.

모든 문서의 기본 원칙 — 1순위는 최적의 결과물입니다. 토큰(비용) 절약은 품질을 해치지 않는 범위의 2순위 부가 효과입니다.

2문서 4종 — 각각 무엇을 하나

00은 입구, 01은 기반, 02·03은 강화

00 통합-설치

선택형 설치 메뉴 (시작점)

01~03을 목록으로 보여주고, 체크한 것만 설치합니다. 이미 설치된 게 있으면 "유지/재구성/교체"를 물어봅니다.

이 파일 하나만 들고 가도 됩니다 — 나머지 문서는 GitHub에서 알아서 가져옵니다.

01 통합구성

기반 공사 (가장 중요)

① 글로벌 셋업: 행동 규칙, 시크릿(.env) 읽기 차단, 자동 알림 훅.

② 프로젝트 셋업: repo 구조를 판별해 작업 기록 체계(docs/PROGRESS·DECISIONS·PROJECT_PLAN)와 /resume·/wrap 명령을 만듭니다.

→ 효과: 어제 하던 작업을 오늘 세션이 기억하고, 실수로 비밀 파일을 읽거나 커밋하는 사고를 막습니다.

02 모델분담

비싼 모델·싼 모델 역할 분담

어려운 판단(설계·난이도 분류·리뷰)은 상위 모델, 정해진 대로 만드는 일(정형 구현·테스트 실행)은 하위 모델이 맡는 전문 에이전트 팀(.claude/agents/)을 만듭니다.

→ 효과: 품질은 지키면서 비용이 줄고, 단계 산출물이 문서로 남아 다음 단계가 이어받습니다.

선택형 부가 기능

3상황별 사용법

내 상황에 맞는 한 문장만 고르면 됨

상황Claude에게 할 말일어나는 일
새 PC 셋업"00_통합-설치.md 를 학습해서 이 PC에 적용해줘"글로벌 규칙·보안 설정 + (선택 시) 사용량 HUD 설치
새 프로젝트 시작"00_통합-설치.md 를 학습해서 이 프로젝트에 적용해줘"repo 유형 판별 → 기록 체계 + /resume·/wrap 생성, 개발 프로젝트면 02·03도 목록에서 선택
특정 문서만"02_모델분담-범용-플레이북.md 를 이 프로젝트에 적용해줘"00을 거치지 않고 그 문서 절차만 실행
확장 기능만"03_확장기능-설치-체크리스트.md 를 적용해줘"3종 체크리스트 → 스코프(전체 PC/이 프로젝트)·호출 방식(자동/수동)을 물어보고 설치
덮어쓰기 걱정은 안 해도 됩니다 — 모든 문서는 설치 전에 기존 구성을 먼저 감지하고, 이미 있으면 유지 / 재구성 / 교체를 반드시 물어보게 설계돼 있습니다.

4설치 후 매일 쓰는 법

하루 루틴은 명령 두 개면 충분

아침: git pull → /resume ("어제 어디까지 했더라?"를 Claude가 보고)
작업: 파일은 @경로/파일명 으로 지정 · 작업 요청엔 성공 기준·검증 명령을 함께
마감: /wrap (오늘 작업·결정을 docs/에 자동 기록) → git 커밋·푸시
  • /resume — 미커밋 변경·기록을 읽고 "지난 작업 X, 다음은 Y?" 브리핑
  • /wrap — 오늘 한 일을 PROGRESS에, 내린 결정을 DECISIONS에 기록. 기록 후 커밋까지 해야 다음 세션이 찾습니다
입문자에게 특히 좋은 습관 3가지 — ① 요청할 때 "구현 후 테스트 돌려서 통과 확인해줘"처럼 검증까지 시키기 ② 무관한 작업으로 넘어갈 땐 /clear ③ 같은 문제를 두 번 교정해도 안 되면 /clear 후 배운 것을 반영해 새로 요청.

5자주 묻는 것

처음 보면 궁금한 것들

Q. 문서가 왜 전부 md 파일인가요?

Claude가 읽고 실행하는 용도라서입니다. 사람용 안내는 이 페이지 하나로 충분하게 유지합니다.

Q. 문서 내용을 그냥 믿어도 되나요?

모든 기술 서술은 공식 문서(code.claude.com/docs)를 당일 조회해 검증한 날짜와 함께 기록됩니다. 각 문서 상단의 버전 표와 "기준: Claude Code v2.1.x" 표기를 확인하세요. Claude Code는 매주 바뀌므로 문서에도 6개월 주기 재검증 체크리스트(§L)가 들어 있습니다.

Q. 팀 프로젝트에도 쓸 수 있나요?

네. 01이 repo 구조(1인 단일 repo ~ 여러 팀원 동시 편집)를 판별해 필요한 만큼만 구성합니다. 기록 파일은 append 전용 + 자동 병합(merge=union)이라 동시 작업 충돌도 자동 처리됩니다.

Q. 시크릿(.env, 키 파일)은 안전한가요?

01의 글로벌 셋업이 .env·인증서·키 파일을 읽기·출력 차단(deny 규칙)합니다. 다만 근본 원칙은 "진짜 비밀은 repo에 두지 않기"입니다.
