## §1. 진단 요약 — 한눈에

결론: 두 문서(00·01)는 정확하다. 다만 "공식 1순위 원칙" 몇 개가 빠져 있었다.

2026-07-20 기준 공식 문서(best-practices · common-workflows · worktrees · headless · mcp)와 전면 대조한 결과, **두 문서에 틀린 내용은 없다**(당일 재검증 완료). 부족한 것은 "커버 범위"다.

| 문서 | 잘 갖춘 것 | 빠진 것 (공식 권장 기준) | 보완 우선순위 |
| --- | --- | --- | --- |
| **00 통합구성** | 세션 기록 체계(/resume·/wrap), 환경 판별, 보안 deny, effort | ① 검증 루프 ② plan mode 워크플로 ③ 컨텍스트 관리 습관 ④ 훅의 "강제 게이트" 활용 ⑤ MCP | ①②③ = 높음 |
| **01 모델분담** | 난이도 라우팅, 검증 스테이지, 문서 핸드오프, 에이전트 로스터 | ① worktree 병렬 실행 ② 헤드리스 팬아웃(CI·대량 마이그레이션) ③ Writer/Reviewer 세션 분리 ④ 인터뷰→스펙 패턴 | ①② = 높음 |

> 💡 **큰 그림** — 공식 best practices의 대전제는 **"컨텍스트 창은 금방 차고, 차면 성능이 떨어진다"**. 00·01은 "세션 간 연속성"(기록)과 "모델 간 분담"(라우팅)을 잘 다루지만, **세션 안에서의 컨텍스트 관리**와 **Claude가 스스로 검증하게 만드는 법**이 빠져 있었다. §4~§6이 그 공백을 채운다.

## §3. 01 모델분담 — 공백과 보완 방향

"모델 축" 분담은 완성형. "세션 축" 분담(병렬·자동화)을 더하면 완전체.

| # | 공백 | 왜 중요한가 | 보완 방향 |
| --- | --- | --- | --- |
| 1 | **worktree 병렬 실행** | 01의 병렬은 "슬라이스를 잘게" 한 줄뿐이었음. 실제 병렬 도구는 git worktree(`claude --worktree`)와 `isolation: worktree`(서브에이전트 격리) | §7 참고. implementer 에이전트에 `isolation: worktree`를 달면 여러 모듈 동시 구현 시 파일 충돌 원천 차단 → **01 v1.3에 반영됨** |
| 2 | **헤드리스 팬아웃** | 대량 정형 전환(01의 주 무대)은 세션 안 서브에이전트보다 `claude -p` 루프가 더 통제 가능(파일당 독립 컨텍스트 + 권한 스코프) | §8 참고. "파일 목록 생성 → 2~3개로 프롬프트 튜닝 → 전체 실행" 3단계 → **01 v1.3에 노트 반영됨** |
| 3 | **Writer/Reviewer 세션 분리** | 01의 reviewer는 서브에이전트. 공식 가이드는 한 발 더: **자기가 짠 코드엔 편향이 생기므로 리뷰는 새 컨텍스트에서** | reviewer를 별도 세션(또는 fresh 서브에이전트)으로 돌리고 diff와 기준만 전달 (§7 패턴 참고) |
| 4 | **인터뷰→스펙 패턴** | 01의 "문서 고정 핸드오프"의 상류 공정. 큰 기능은 Claude가 먼저 **나를 인터뷰**하게 해 SPEC.md를 만들고, 새 세션에서 실행 | §5 응용 참고. 프롬프트: "이 기능을 만들고 싶다. AskUserQuestion으로 나를 인터뷰해서 SPEC.md를 완성하라" |
| 5 | **리뷰어 과잉 지적 주의 없음** | 공식 경고: "빈틈을 찾으라고 하면 멀쩡한 코드에서도 찾아낸다" → 다 반영하면 과설계 | reviewer 프롬프트에 "정확성·요구사항에 영향 주는 것만, 스타일 제외" 명시 → **01 v1.3 reviewer에 반영됨** |

> 💡 **01과 궁합이 좋은 신기능** — ① `/code-review`: 번들 스킬. 현재 diff를 fresh 서브에이전트가 버그 관점으로 리뷰(01의 reviewer 초안으로 바로 사용 가능). ② `/goal`: "테스트 전부 통과"처럼 완료 조건을 걸면 조건이 성립할 때까지 턴을 이어감. ③ 에이전트 팀·백그라운드 서브에이전트(v2.1.198+ 기본): 여러 서브에이전트가 돌아가는 동안 메인이 계속 일함.

## §4. 검증 루프 — 가장 먼저 익힐 습관

"Claude가 스스로 확인할 방법"을 주면, 지켜보는 세션이 맡기는 세션이 된다.

Claude는 "다 된 것처럼 보이면" 멈춘다. 통과/실패 신호를 주지 않으면 **당신이 검증 루프**가 되어 모든 실수를 눈으로 잡아야 한다. 테스트·빌드 종료코드·린터·스크린샷 비교, 뭐든 Claude가 읽을 수 있는 신호를 줄 것.

| 전략 | ❌ Before | ✅ After |
| --- | --- | --- |
| 검증 기준 제공 | "이메일 검증 함수 만들어줘" | "validateEmail 함수 작성. 테스트 케이스: user@example.com→true, invalid→false, user@.com→false. **구현 후 테스트 실행해서 통과 확인**" |
| UI는 눈으로 검증 | "대시보드 예쁘게" | "[스크린샷 붙여넣기] 이 디자인으로 구현하고, **결과를 스크린샷 찍어 원본과 비교**해서 차이를 고쳐라" |
| 증상 아닌 원인 | "빌드가 깨져" | "빌드가 이 에러로 실패: [에러 붙여넣기]. **근본 원인을 고치고 빌드 성공을 확인**하라. 에러 숨기기 금지" |

**강도별 — 검증을 얼마나 세게 걸까**
- **프롬프트 안에서**: "구현 후 테스트 돌리고 실패하면 고쳐" — 오늘부터 바로 가능
- **세션 전체에**: `/goal`로 완료 조건 설정 — 매 턴 후 별도 평가자가 재확인, 성립까지 계속 작업
- **결정적 게이트로**: Stop 훅이 체크 스크립트를 돌려 **통과 전엔 턴 종료 자체를 차단** (8회 연속 차단 시 해제)
- **제3자 검증으로**: fresh 서브에이전트가 결과 반박 시도 — 짠 사람이 채점하지 않게

> ✅ **증거를 요구하라** — "성공했다"는 말 대신 테스트 출력·실행한 명령과 결과·스크린샷을 보여달라고 할 것. 증거 확인이 재검증보다 빠르고, 자리 비운 세션에도 통한다.

## §5. 탐색 → 계획 → 구현 → 커밋

코딩부터 시키면 엉뚱한 문제를 푼다. plan mode로 조사와 실행을 분리.

```text
1. 탐색 (plan mode) — "src/auth 읽고 세션 처리 방식 파악해" (파일 수정 없음)
2. 계획 (plan mode) — "Google OAuth 추가하려면 뭘 바꿔야 해? 계획 세워"
                       → Ctrl+G로 계획을 에디터에서 직접 수정
3. 구현 (일반 모드) — "계획대로 구현하고 테스트 작성·실행, 실패 수정"
4. 커밋 — "설명적인 메시지로 커밋하고 PR 열어"
```

- plan mode 진입: `Shift+Tab`으로 순환(default → acceptEdits → plan) 또는 `claude --permission-mode plan`으로 시작. 상태바에 `⏸ plan mode on` 표시.
- plan mode에선 Claude가 읽기와 제안만 하고 **승인 전까지 디스크를 건드리지 않는다.**

> ⚠️ **계획이 항상 정답은 아님** — 오타 수정·로그 한 줄·변수 이름 바꾸기처럼 **diff를 한 문장으로 설명할 수 있으면 계획을 생략**하고 바로 시킬 것. 계획이 유용한 건 접근이 불확실하거나, 여러 파일을 건드리거나, 낯선 코드일 때다.

**응용 — 큰 기능은 인터뷰부터** (프롬프트 템플릿)

```text
[기능 설명]을 만들고 싶다. AskUserQuestion 도구로 나를 상세히 인터뷰하라.
기술 구현·UI/UX·엣지케이스·트레이드오프를 물어보되, 뻔한 질문 말고
내가 미처 생각 못 한 어려운 부분을 파고들어라.
전부 다뤘으면 완전한 스펙을 SPEC.md로 작성하라.
```

스펙이 완성되면 **새 세션**에서 실행 — 깨끗한 컨텍스트가 구현에만 집중하고, 스펙 문서가 남는다(01의 "문서 고정 핸드오프"와 동일 철학).

## §6. 컨텍스트 관리 — 성능 저하의 주범 잡기

컨텍스트 창이 차면 Claude는 앞의 지시를 "잊기" 시작한다.

| 도구 | 언제 | 효과 |
| --- | --- | --- |
| `/clear` | **무관한 작업으로 넘어갈 때마다** | 컨텍스트 완전 초기화. 가장 강력하고 가장 자주 써야 할 명령 |
| `/compact 지시문` | 같은 작업을 이어가며 압축할 때 | 예: `/compact API 변경 내용 위주로` — 남길 것을 지정 |
| `/context` | 뭐가 로드됐는지 궁금할 때 | CLAUDE.md·rules·스킬이 실제로 읽혔는지 확인 |
| `Esc Esc` / `/rewind` | 방향이 틀어졌을 때 | 체크포인트로 대화·코드 상태 복원, 또는 구간만 요약 |
| `/btw 질문` | 본 작업과 무관한 짧은 질문 | 답이 오버레이로 뜨고 **대화 기록에 남지 않음** — 컨텍스트 오염 zero |
| 서브에이전트 위임 | 넓은 코드 조사 | "서브에이전트로 토큰 갱신 로직 조사해줘" — 파일 읽기는 격리 컨텍스트에서, 요약만 돌아옴 |

> 🚫 **2회 교정 룰** — 같은 문제로 Claude를 **두 번 넘게 교정했다면** 컨텍스트가 실패한 시도들로 오염된 상태다. 계속 교정하지 말고 `/clear` 후 배운 내용을 반영한 더 나은 프롬프트로 새로 시작할 것 — 거의 항상 이쪽이 빠르다.

> 💡 **체크포인트로 과감해지기** — 매 프롬프트마다 체크포인트가 생기고 파일 스냅샷도 저장된다. "위험해 보이는 접근을 일단 시켜보고, 안 되면 `/rewind`"가 가능해진다. 단, Claude의 편집 도구로 바꾼 것만 추적하므로 **git 대체가 아니다.**

## §7. 병렬 실행 — git worktree

터미널 2개, 세션 2개, 충돌 0 — 01 모델분담의 빠진 조각(→ 01 v1.3에 반영됨).

worktree는 같은 저장소를 **다른 폴더·다른 브랜치**로 체크아웃한 사본이다. 세션마다 하나씩 쓰면 한쪽이 기능을 만들고 다른 쪽이 버그를 고쳐도 파일이 절대 충돌하지 않는다.

```bash
# 터미널 1 — 기능 개발 (worktree: .claude/worktrees/feature-auth, 브랜치: worktree-feature-auth)
claude --worktree feature-auth

# 터미널 2 — 버그 수정 (이름만 다르게)
claude --worktree bugfix-login

# PR 브랜치에서 바로 시작 (따옴표 필수 — #이 셸 주석이 되지 않게)
claude --worktree "#1234"
```

- **정리**: 세션 종료 시 작업물이 있으면 유지/삭제를 물어봄. 깨끗하면 자동 삭제.
- **.env 자동 복사**: worktree는 새 체크아웃이라 gitignore된 파일이 없음 → 프로젝트 루트에 `.worktreeinclude` 파일을 만들고 `.env` 등을 나열하면 자동 복사.
- **.gitignore에 추가**: `.claude/worktrees/` 를 넣어 untracked 노이즈 방지.
- **서브에이전트 격리**: 에이전트 frontmatter에 `isolation: worktree` 한 줄 — 01의 implementer가 여러 모듈을 동시에 고칠 때 충돌 원천 차단.

**패턴 — Writer / Reviewer 세션 분리 리뷰**

| 세션 A (작성자) | 세션 B (리뷰어) |
| --- | --- |
| "API 엔드포인트에 rate limiter 구현해" | |
| | "@src/middleware/rateLimiter.ts 리뷰. 엣지케이스·경쟁 조건·기존 미들웨어 패턴과의 일관성 확인" |
| "리뷰 피드백: [세션 B 출력]. 반영해" | |

> 💡 **왜 세션을 나누나** — 방금 자기가 짠 코드에는 편향이 생긴다. 새 컨텍스트의 리뷰어는 diff와 기준만 보고 판단한다. 같은 원리로 "한 세션이 테스트 작성 → 다른 세션이 통과 코드 작성"도 가능.

## §8. 자동화 — 헤드리스 모드와 CI

`claude -p` 하나로 린터·리뷰어·대량 마이그레이션 러너가 된다.

```bash
# 일회성 질문 (비대화형)
claude -p "이 프로젝트가 뭐 하는지 설명해"

# 파이프로 데이터 주입
cat build-error.txt | claude -p "이 빌드 에러의 근본 원인을 간결히 설명해"

# CI·스크립트용 — --bare: 훅·스킬·MCP·CLAUDE.md 자동 로드 생략 (어느 머신에서나 동일 결과)
claude --bare -p "이 파일 요약해" --allowedTools "Read"

# 구조화된 출력 (세션ID·비용 포함 JSON)
claude -p "API 엔드포인트 전부 나열" --output-format json
```

**레시피 — 프로젝트 전용 린터 만들기** (package.json scripts)

```json
{
  "scripts": {
    "lint:claude": "git diff main | claude -p \"너는 오타 린터다. 이 diff의 오타마다 파일:줄 한 줄, 문제 한 줄로 보고. 그 외엔 아무것도 출력하지 마.\""
  }
}
```

**레시피 — 대량 마이그레이션 팬아웃** (01 모델분담의 실행 엔진 · 3단계: 목록 → 튜닝 → 전체 실행)

```bash
# 1) Claude에게 대상 파일 목록 생성시키기 → files.txt
# 2) 2~3개 파일로 먼저 돌려 프롬프트를 다듬은 뒤
# 3) 전체 실행 — --allowedTools로 권한을 좁게 (무인 실행의 안전벨트)
for file in $(cat files.txt); do
  claude -p "$file 을 React에서 Vue로 마이그레이션. OK 또는 FAIL만 출력." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

- **권한 문법**: `Bash(git diff *)`의 **공백+\***이 중요 — 없으면 `git diff-index`까지 매칭된다.
- **대화 이어가기**: `--continue`(가장 최근) / `--resume 세션ID`(특정). JSON 출력의 `session_id`를 잡아두면 스크립트에서 후속 질문 가능.
- **무인 실행 모드**: `--permission-mode auto` — 분류기가 위험한 명령만 차단하고 나머지는 통과. 잠근 CI엔 `dontAsk`.
- **정기 실행**: 아침 PR 리뷰·주간 의존성 감사는 Routines(클라우드)·GitHub Actions(CI)·데스크톱 예약 작업 중 택1.

## §9. MCP — 외부 도구 직접 연결

이슈트래커·DB·모니터링을 복붙 대신 Claude가 직접 읽고 쓰게.

다른 툴의 내용을 채팅에 복사해 넣고 있다면 그게 MCP를 붙일 신호다. 연결하면 이런 게 한 문장이 된다:
- "JIRA 이슈 ENG-4521에 적힌 기능을 구현하고 GitHub에 PR 올려"
- "Sentry에서 이 기능 관련 에러 확인해"
- "PostgreSQL에서 이 기능을 쓴 유저 10명의 이메일 뽑아줘"

```bash
# 원격 서버 (예: Notion)
claude mcp add --transport http notion https://mcp.notion.com/mcp

# 로컬 서버 (예: Airtable) — "--" 뒤는 서버 실행 명령
claude mcp add --env AIRTABLE_API_KEY=YOUR_KEY --transport stdio airtable \
  -- npx -y airtable-mcp-server

# 상태 확인 / 인증
/mcp
```

- **스코프 3종**: 기본은 개인(local). `--scope project`로 추가하면 `.mcp.json`에 저장돼 **팀과 커밋·공유**(각자 첫 실행 때 승인). `--scope user`는 내 모든 프로젝트.
- **리소스 참조**: `@server:resource` 형식 — 예: `@github:repos/owner/repo/issues`.
- **검증된 서버 찾기**: Anthropic Directory(claude.ai/directory)의 커넥터는 전부 `claude mcp add`로 추가 가능.

> 🚫 **보안 — 신뢰하는 서버만** — 외부 콘텐츠를 가져오는 MCP 서버는 **프롬프트 인젝션** 경로가 될 수 있다(가져온 텍스트 안에 악성 지시가 숨는 공격). 출처가 확실한 서버만 연결하고, 프로젝트 `.mcp.json`에 들어온 서버는 승인 전 내용을 확인할 것. 00 문서의 시크릿 원칙과 동일하게 — API 키는 `--env`나 환경변수로, 절대 커밋 금지.

## §A. 흔한 실패 패턴 5 — 미리 알면 시간 절약

공식 가이드가 명시한 안티패턴. 신입이 전부 한 번씩 겪는 것들.

| 패턴 | 증상 | 처방 |
| --- | --- | --- |
| 싱크대 세션 | 작업 A 하다가 무관한 B 물어보고 다시 A로 — 컨텍스트가 잡동사니 | 무관한 작업 전 `/clear` |
| 무한 교정 | 틀림→교정→또 틀림→또 교정… 실패한 시도가 컨텍스트에 쌓임 | 교정 2회 실패 시 `/clear` + 배운 것 반영한 새 프롬프트 |
| 비대한 CLAUDE.md | 규칙이 많아질수록 오히려 안 지킴 — 중요한 규칙이 소음에 묻힘 | 각 줄마다 "지우면 실수하나?" 질문. 아니면 삭제하거나 훅으로 전환 |
| 검증 없는 신뢰 | 그럴듯한 구현이 엣지케이스에서 무너짐 | 테스트·스크립트·스크린샷 없으면 배포 금지 (§4) |
| 무한 탐색 | 범위 없이 "조사해봐" → 수백 파일 읽으며 컨텍스트 소진 | 조사 범위를 좁히거나 서브에이전트에 위임 |

## §C. 신입 3주 도입 로드맵

한꺼번에 다 하지 말 것 — 습관 하나씩.

**1주차 — 기본기 (00 문서 + 검증 습관)**
- [ ] 00 문서대로 글로벌·프로젝트 셋업 (/resume·/wrap 루틴 시작)
- [ ] `/init`으로 CLAUDE.md 초안 만들고 다듬기
- [ ] 모든 작업 요청에 **성공 기준 + 검증 명령** 포함하는 습관 (§4)
- [ ] 무관한 작업 전 `/clear`, 교정 2회 실패 시 `/clear` (§6)

**2주차 — 워크플로 (계획과 되감기)**
- [ ] 여러 파일 건드리는 작업은 plan mode(Shift+Tab)로 시작 (§5)
- [ ] 방향 틀어지면 `Esc Esc` 되감기 활용
- [ ] 넓은 조사는 "서브에이전트로 조사해줘"로 위임
- [ ] 구현 끝나면 `/code-review` 한 번 돌리기

**3주차 — 확장 (01 문서 + 병렬·자동화·연결)**
- [ ] 01 문서 적용 — 난이도 rubric + 에이전트 로스터 생성
- [ ] `claude --worktree`로 병렬 세션 1회 경험 (§7)
- [ ] 팀이 쓰는 외부 도구 1개 MCP 연결 (§9 — 보안 주의 먼저 읽기)
- [ ] 반복 작업 하나를 `claude -p` 스크립트로 (§8)

> 💡 **마지막 조언 — 직관 기르기** — 이 가이드의 패턴은 출발점이지 법칙이 아니다. 깊은 문제 하나를 파는 중이면 컨텍스트를 일부러 쌓는 게 맞고, 탐색적인 작업이면 계획 없이 맡기는 게 맞을 때도 있다. 잘 됐을 때 내가 뭘 했는지(프롬프트 구조·제공한 컨텍스트·모드)를 관찰하는 습관이 어떤 가이드보다 오래 간다.
