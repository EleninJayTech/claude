00 통합구성 — 공백과 보완 방향

기록 체계는 완성형. 세션 "안"의 운영 기법을 더하면 됨

#공백왜 중요한가보완 방향
1검증 루프 부재공식 가이드 1순위: "Claude에게 스스로 돌릴 수 있는 체크(테스트·빌드·스크린샷)를 줘라". 이게 없으면 사람이 검증 루프가 됨§4 참고. F-1 규칙 블록에 "작업 요청 시 성공 기준·검증 명령을 함께 준다" 한 줄 추가 권장
2plan mode 미소개공식 권장 기본 워크플로(탐색→계획→구현→커밋)의 축. 00엔 opusplan 언급만 있음§5 참고. Shift+Tab 순환·Ctrl+G 계획 편집을 매일 루틴(§G)에 편입
3컨텍스트 관리 습관 부재/clear·/compact 지시·/context·체크포인트(/rewind)가 없음. PreCompact 훅(알림)만 있음§6 참고. "무관한 작업 전 /clear"를 매일 루틴에 추가
4훅을 알림용으로만 사용훅의 진짜 힘은 강제 게이트: CLAUDE.md는 권고지만 훅은 예외 없이 실행됨PostToolUse로 편집 후 린트 자동 실행, PreToolUse로 금지 폴더 쓰기 차단. "훅 써달라"고 Claude에게 부탁하면 만들어 줌
5MCP 전무이슈트래커·DB·모니터링을 직접 읽고 쓰는 표준 통로인데 문서 어디에도 없음§9 참고. 프로젝트 .mcp.json은 커밋 대상이므로 00의 "공유 vs 개인" 표에 추가할 가치
6/init 미언급CLAUDE.md를 처음 만들 때 공식 시작점(코드베이스 분석해 초안 생성)§E 프로젝트 셋업 첫 단계로 "/init 실행 후 다듬기" 추가 권장
유지할 것git-status 우선 재개, append 전용 + merge=union, 방법 A(교차 기록), deny 보안 체계는 공식 문서와 대조해도 손댈 곳이 없습니다. CLAUDE.md "30~200줄, 행동규칙과 사실 분리" 지침도 공식 권장("짧게, 각 줄마다 '지우면 실수하나?' 질문")과 일치합니다.

01 모델분담 — 공백과 보완 방향

"모델 축" 분담은 완성형. "세션 축" 분담(병렬·자동화)을 더하면 완전체

#공백왜 중요한가보완 방향
1worktree 병렬 실행01의 병렬은 "슬라이스를 잘게" 한 줄뿐. 실제 병렬 도구는 git worktree(claude --worktree)와 isolation: worktree(서브에이전트 격리)§7 참고. implementer 에이전트에 isolation: worktree를 달면 여러 모듈 동시 구현 시 파일 충돌 원천 차단
2헤드리스 팬아웃대량 정형 전환(01의 주 무대)은 세션 안 서브에이전트보다 claude -p 루프가 더 통제 가능(파일당 독립 컨텍스트 + 권한 스코프)§8 참고. "파일 목록 생성 → 2~3개로 프롬프트 튜닝 → 전체 실행" 3단계
3Writer/Reviewer 세션 분리01의 reviewer는 서브에이전트. 공식 가이드는 한 발 더: 자기가 짠 코드엔 편향이 생기므로 리뷰는 새 컨텍스트에서reviewer를 별도 세션(또는 fresh 서브에이전트)으로 돌리고 diff와 기준만 전달
4인터뷰→스펙 패턴01의 "문서 고정 핸드오프"의 상류 공정. 큰 기능은 Claude가 먼저 나를 인터뷰하게 해 SPEC.md를 만들고, 새 세션에서 실행프롬프트: "이 기능을 만들고 싶다. AskUserQuestion으로 나를 인터뷰해서 SPEC.md를 완성하라"
5리뷰어 과잉 지적 주의 없음공식 경고: "빈틈을 찾으라고 하면 멀쩡한 코드에서도 찾아낸다" → 다 반영하면 과설계reviewer 프롬프트에 "정확성·요구사항에 영향 주는 것만, 스타일 제외" 명시 (01 reviewer 정의에 이미 유사 취지 있음 — 강화 권장)
01과 궁합이 좋은 신기능① /code-review — 번들 스킬. 현재 diff를 fresh 서브에이전트가 버그 관점으로 리뷰(01의 reviewer 초안으로 바로 사용 가능). ② /goal — "테스트 전부 통과"처럼 완료 조건을 걸면 조건이 성립할 때까지 턴을 이어감. ③ 에이전트 팀·백그라운드 서브에이전트(v2.1.198+ 기본) — 여러 서브에이전트가 돌아가는 동안 메인이 계속 일함.

검증 루프 — 가장 먼저 익힐 습관

"Claude가 스스로 확인할 방법"을 주면, 지켜보는 세션이 맡기는 세션이 된다

Claude는 "다 된 것처럼 보이면" 멈춥니다. 통과/실패 신호를 주지 않으면 당신이 검증 루프가 되어 모든 실수를 눈으로 잡아야 합니다. 테스트·빌드 종료코드·린터·스크린샷 비교, 뭐든 Claude가 읽을 수 있는 신호를 주세요.

전략❌ Before✅ After
검증 기준 제공"이메일 검증 함수 만들어줘""validateEmail 함수 작성. 테스트 케이스: user@example.com→true, invalid→false, user@.com→false. 구현 후 테스트 실행해서 통과 확인"
UI는 눈으로 검증"대시보드 예쁘게""[스크린샷 붙여넣기] 이 디자인으로 구현하고, 결과를 스크린샷 찍어 원본과 비교해서 차이를 고쳐라"
증상 아닌 원인"빌드가 깨져""빌드가 이 에러로 실패: [에러 붙여넣기]. 근본 원인을 고치고 빌드 성공을 확인하라. 에러 숨기기 금지"

강도별검증을 얼마나 세게 걸까

  • 프롬프트 안에서: "구현 후 테스트 돌리고 실패하면 고쳐" — 오늘부터 바로 가능
  • 세션 전체에: /goal로 완료 조건 설정 — 매 턴 후 별도 평가자가 재확인, 성립까지 계속 작업
  • 결정적 게이트로: Stop 훅이 체크 스크립트를 돌려 통과 전엔 턴 종료 자체를 차단 (8회 연속 차단 시 해제)
  • 제3자 검증으로: fresh 서브에이전트가 결과 반박 시도 — 짠 사람이 채점하지 않게
증거를 요구하세요"성공했다"는 말 대신 테스트 출력·실행한 명령과 결과·스크린샷을 보여달라고 하세요. 증거 확인이 재검증보다 빠르고, 자리 비운 세션에도 통합니다.

탐색 → 계획 → 구현 → 커밋

코딩부터 시키면 엉뚱한 문제를 풉니다. plan mode로 조사와 실행을 분리

1. 탐색 (plan mode) — "src/auth 읽고 세션 처리 방식 파악해" (파일 수정 없음)
2. 계획 (plan mode) — "Google OAuth 추가하려면 뭘 바꿔야 해? 계획 세워" → Ctrl+G로 계획을 에디터에서 직접 수정
3. 구현 (일반 모드) — "계획대로 구현하고 테스트 작성·실행, 실패 수정"
4. 커밋 — "설명적인 메시지로 커밋하고 PR 열어"
  • plan mode 진입: Shift+Tab으로 순환(default → acceptEdits → plan) 또는 claude --permission-mode plan으로 시작. 상태바에 ⏸ plan mode on 표시.
  • plan mode에선 Claude가 읽기와 제안만 하고 승인 전까지 디스크를 건드리지 않습니다.
계획이 항상 정답은 아님오타 수정·로그 한 줄·변수 이름 바꾸기처럼 diff를 한 문장으로 설명할 수 있으면 계획을 생략하고 바로 시키세요. 계획이 유용한 건 접근이 불확실하거나, 여러 파일을 건드리거나, 낯선 코드일 때입니다.

응용큰 기능은 인터뷰부터

프롬프트 템플릿 — 인터뷰 → 스펙
[기능 설명]을 만들고 싶다. AskUserQuestion 도구로 나를 상세히 인터뷰하라.
기술 구현·UI/UX·엣지케이스·트레이드오프를 물어보되, 뻔한 질문 말고
내가 미처 생각 못 한 어려운 부분을 파고들어라.
전부 다뤘으면 완전한 스펙을 SPEC.md로 작성하라.

스펙이 완성되면 새 세션에서 실행하세요 — 깨끗한 컨텍스트가 구현에만 집중하고, 스펙 문서가 남습니다(01의 "문서 고정 핸드오프"와 동일 철학).

컨텍스트 관리 — 성능 저하의 주범 잡기

컨텍스트 창이 차면 Claude는 앞의 지시를 "잊기" 시작한다

도구언제효과
/clear무관한 작업으로 넘어갈 때마다컨텍스트 완전 초기화. 가장 강력하고 가장 자주 써야 할 명령
/compact 지시문같은 작업을 이어가며 압축할 때예: /compact API 변경 내용 위주로 — 남길 것을 지정
/context뭐가 로드됐는지 궁금할 때CLAUDE.md·rules·스킬이 실제로 읽혔는지 확인
Esc Esc / /rewind방향이 틀어졌을 때체크포인트로 대화·코드 상태 복원, 또는 구간만 요약
/btw 질문본 작업과 무관한 짧은 질문답이 오버레이로 뜨고 대화 기록에 남지 않음 — 컨텍스트 오염 zero
서브에이전트 위임넓은 코드 조사"서브에이전트로 토큰 갱신 로직 조사해줘" — 파일 읽기는 격리 컨텍스트에서, 요약만 돌아옴
2회 교정 룰같은 문제로 Claude를 두 번 넘게 교정했다면 컨텍스트가 실패한 시도들로 오염된 상태입니다. 계속 교정하지 말고 /clear 후 배운 내용을 반영한 더 나은 프롬프트로 새로 시작하세요 — 거의 항상 이쪽이 빠릅니다.
체크포인트로 과감해지기매 프롬프트마다 체크포인트가 생기고 파일 스냅샷도 저장됩니다. "위험해 보이는 접근을 일단 시켜보고, 안 되면 /rewind"가 가능해집니다. 단, Claude의 편집 도구로 바꾼 것만 추적하므로 git 대체가 아닙니다.

병렬 실행 — git worktree

터미널 2개, 세션 2개, 충돌 0 — 01 모델분담의 빠진 조각

worktree는 같은 저장소를 다른 폴더·다른 브랜치로 체크아웃한 사본입니다. 세션마다 하나씩 쓰면 한쪽이 기능을 만들고 다른 쪽이 버그를 고쳐도 파일이 절대 충돌하지 않습니다.

터미널 1 / 터미널 2
# 터미널 1 — 기능 개발 (worktree: .claude/worktrees/feature-auth, 브랜치: worktree-feature-auth)
claude --worktree feature-auth

# 터미널 2 — 버그 수정 (이름만 다르게)
claude --worktree bugfix-login

# PR 브랜치에서 바로 시작 (따옴표 필수 — #이 셸 주석이 되지 않게)
claude --worktree "#1234"
  • 정리: 세션 종료 시 작업물이 있으면 유지/삭제를 물어봅니다. 깨끗하면 자동 삭제.
  • .env 자동 복사: worktree는 새 체크아웃이라 gitignore된 파일이 없음 → 프로젝트 루트에 .worktreeinclude 파일을 만들고 .env 등을 나열하면 자동 복사.
  • .gitignore에 추가: .claude/worktrees/ 를 넣어 untracked 노이즈 방지.
  • 서브에이전트 격리: 에이전트 frontmatter에 isolation: worktree 한 줄 — 01의 implementer가 여러 모듈을 동시에 고칠 때 충돌 원천 차단.

패턴Writer / Reviewer — 세션 분리 리뷰

세션 A (작성자)세션 B (리뷰어)
"API 엔드포인트에 rate limiter 구현해"
"@src/middleware/rateLimiter.ts 리뷰. 엣지케이스·경쟁 조건·기존 미들웨어 패턴과의 일관성 확인"
"리뷰 피드백: [세션 B 출력]. 반영해"
왜 세션을 나누나방금 자기가 짠 코드에는 편향이 생깁니다. 새 컨텍스트의 리뷰어는 diff와 기준만 보고 판단합니다. 같은 원리로 "한 세션이 테스트 작성 → 다른 세션이 통과 코드 작성"도 가능합니다.

claude -p "API 엔드포인트 전부 나열" --output-format json

레시피대량 마이그레이션 팬아웃 — 01 모델분담의 실행 엔진

3단계: 목록 → 튜닝 → 전체 실행
# 1) Claude에게 대상 파일 목록 생성시키기 → files.txt
# 2) 2~3개 파일로 먼저 돌려 프롬프트를 다듬은 뒤
# 3) 전체 실행 — --allowedTools로 권한을 좁게 (무인 실행의 안전벨트)
for file in $(cat files.txt); do
  claude -p "$file 을 React에서 Vue로 마이그레이션. OK 또는 FAIL만 출력." \
    --allowedTools "Edit,Bash(git commit *)"
done
  • 권한 문법: Bash(git diff *)의 공백+*이 중요 — 없으면 git diff-index까지 매칭됩니다.
  • 대화 이어가기: --continue(가장 최근) / --resume 세션ID(특정). JSON 출력의 session_id를 잡아두면 스크립트에서 후속 질문 가능.
  • 무인 실행 모드: --permission-mode auto — 분류기가 위험한 명령만 차단하고 나머지는 통과. 잠근 CI엔 dontAsk.
  • 정기 실행: 아침 PR 리뷰·주간 의존성 감사는 Routines(클라우드)·GitHub Actions(CI)·데스크톱 예약 작업 중 택1.

MCP — 외부 도구 직접 연결

이슈트래커·DB·모니터링을 복붙 대신 Claude가 직접 읽고 쓰게

다른 툴의 내용을 채팅에 복사해 넣고 있다면 그게 MCP를 붙일 신호입니다. 연결하면 이런 게 한 문장이 됩니다:

  • "JIRA 이슈 ENG-4521에 적힌 기능을 구현하고 GitHub에 PR 올려"
  • "Sentry에서 이 기능 관련 에러 확인해"
  • "PostgreSQL에서 이 기능을 쓴 유저 10명의 이메일 뽑아줘"
서버 추가 — 원격(HTTP)과 로컬(stdio)
# 원격 서버 (예: Notion)
claude mcp add --transport http notion https://mcp.notion.com/mcp

# 로컬 서버 (예: Airtable) — "--" 뒤는 서버 실행 명령
claude mcp add --env AIRTABLE_API_KEY=YOUR_KEY --transport stdio airtable \
  -- npx -y airtable-mcp-server

# 상태 확인 / 인증
/mcp
  • 스코프 3종: 기본은 개인(local). --scope project로 추가하면 .mcp.json에 저장돼 팀과 커밋·공유(각자 첫 실행 때 승인). --scope user는 내 모든 프로젝트.
  • 리소스 참조: @server:resource 형식 — 예: @github:repos/owner/repo/issues.
  • 검증된 서버 찾기: Anthropic Directory(claude.ai/directory)의 커넥터는 전부 claude mcp add로 추가 가능.
보안 — 신뢰하는 서버만외부 콘텐츠를 가져오는 MCP 서버는 프롬프트 인젝션 경로가 될 수 있습니다(가져온 텍스트 안에 악성 지시가 숨는 공격). 출처가 확실한 서버만 연결하고, 프로젝트 .mcp.json에 들어온 서버는 승인 전 내용을 확인하세요. 00 문서의 시크릿 원칙과 동일하게 — API 키는 --env나 환경변수로, 절대 커밋 금지.
